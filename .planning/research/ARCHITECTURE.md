# ARCHITECTURE — Cloudflare Tunnel + Access + Tailscale for `jarvis`

**Scope.** Securely expose Lumio (Supabase stack) from a home mini-PC
(`jarvis`, Ubuntu 26.04, single dev) using **Cloudflare Tunnel** for the
public app surface and **Tailscale + Cloudflare Access** for the admin
surface (Supabase Studio, internal dashboards).

**Stance.** Security wins ties. The design assumes:

- One named tunnel per host (`jarvis-tunnel`), multiple ingress rules.
- Admin services (Studio, Kong admin API, Postgres) are NEVER published
  over the tunnel. They sit behind Tailscale; CF Access is layered on
  top only when an admin surface must be browser-reachable from a
  non-tailnet device.
- Cloudflare can see plaintext at the edge for tunnelled HTTP — keep
  truly sensitive admin off the tunnel entirely (see §5).

## 1. Installing `cloudflared` on Ubuntu 26.04 (workaround)

Cloudflare's apt repo (`https://pkg.cloudflare.com/cloudflared`) ships
suites for `focal` (20.04), `jammy` (22.04) and `noble` (24.04) only.
There is no `questing` / 26.04 suite at the time of writing. The
established workaround is to **pin the apt source to `noble`** — the
binary is statically linked Go and runs cleanly on 26.04.

### 1.1 GPG key

```bash
sudo install -d -m 0755 /usr/share/keyrings
curl -fsSL https://pkg.cloudflare.com/cloudflare-main.gpg \
  | sudo tee /usr/share/keyrings/cloudflare-main.gpg >/dev/null
sudo chmod 0644 /usr/share/keyrings/cloudflare-main.gpg
```

> Cloudflare rotated this signing key; the previous keys are deprecated
> and will be **removed on 30 April 2026**. Re-fetch if you set this up
> before mid-2025.

### 1.2 deb822 sources file (preferred on 24.04+)

Create `/etc/apt/sources.list.d/cloudflared.sources`:

```
Types: deb
URIs: https://pkg.cloudflare.com/cloudflared
Suites: noble
Components: main
Architectures: amd64 arm64
Signed-By: /usr/share/keyrings/cloudflare-main.gpg
```

Then:

```bash
sudo apt update
sudo apt install cloudflared
cloudflared --version
```

### 1.3 Why noble and not a manual `.deb`

- The apt-pinned repo gives auto-updates via unattended-upgrades.
- The manual `.deb` from GitHub releases works but you lose update
  automation; acceptable as a fallback if pkg.cloudflare.com is ever
  unavailable for `noble`.

### 1.4 Auth flow: `tunnel login` vs token-only

`cloudflared` distinguishes **management credentials** from **run-time
credentials**:

| File / Token              | Purpose                                | Where it lives                                 |
| ------------------------- | -------------------------------------- | ---------------------------------------------- |
| `~/.cloudflared/cert.pem` | Management cert; creates/deletes/lists tunnels and DNS routes | Generated by `cloudflared tunnel login` |
| `<TUNNEL_UUID>.json`      | Per-tunnel credentials; only authorises running that one tunnel | Generated by `cloudflared tunnel create` |
| Tunnel **token** (JWT)    | Base64-encoded JWT equivalent of the JSON, used with `--token` for remote-managed tunnels | Shown in the Zero Trust dashboard |

**Resolution order at run-time** (highest priority first):
`--token` → `--token-file` → `--credentials-contents` →
`--credentials-file` → search by tunnel UUID.

**Recommendation for `jarvis`** (locally-managed tunnel, single host):

1. Run `cloudflared tunnel login` **once** as your user — produces
   `~/.cloudflared/cert.pem`. Keep this file off the server after
   provisioning (move it to your offline secret store; it's only needed
   when you want to mutate tunnel/DNS config).
2. `cloudflared tunnel create jarvis-tunnel` — emits
   `~/.cloudflared/<UUID>.json`.
3. Move credentials JSON to `/etc/cloudflared/<UUID>.json`,
   `chown root:cloudflared`, `chmod 0640`.
4. Do **not** distribute the runtime token: dashboard-issued tokens are
   the surface attackers go after first; sticking to local-managed +
   credentials-file means the tunnel cannot be hijacked from CF dash if
   your account is compromised without also pulling the JSON.

## 2. Tunnel config: `/etc/cloudflared/config.yml` vs Docker

Two viable shapes:

| Option              | Pros                                                          | Cons                                                                  |
| ------------------- | ------------------------------------------------------------- | --------------------------------------------------------------------- |
| **systemd service** | Survives reboots cleanly; integrates with journald; less moving parts; can reach `127.0.0.1` services directly | Default unit runs as root; needs hardening (§6) |
| **Docker container**| Reproducible image pin; isolated FS                           | Needs `network_mode: host` (or compose-internal DNS) to reach Kong on `localhost:8000`; one more layer to patch |

For a single-host home setup with native services, **systemd is the
right default**. Use Docker only if the rest of the stack is dockerized
and you can put `cloudflared` on the same compose network as Kong.

### 2.1 Install as a service

After step 1.4 above:

```bash
sudo cloudflared --config /etc/cloudflared/config.yml service install
sudo systemctl enable --now cloudflared
```

The installer creates `/etc/systemd/system/cloudflared.service`. Override
it with `systemctl edit cloudflared` (see §6 for the hardening drop-in).

### 2.2 Directory layout

```
/etc/cloudflared/
  config.yml                # ingress + originRequest
  <TUNNEL_UUID>.json        # runtime credentials, 0640 root:cloudflared
/home/toto/jarvis/
  cloudflared-dev/          # ad-hoc dev configs (not used by systemd)
```

- **Production**: `/etc/cloudflared/` is root-owned, group `cloudflared`
  (so the service user can read but not write).
- **Dev/ad-hoc**: a separate config in `~/jarvis/cloudflared-dev/` run
  with `cloudflared tunnel --config … run` for temporary test tunnels —
  keep them on a different tunnel UUID so test config never overwrites
  prod credentials.

### 2.3 Credentials path

The `credentials-file` value **must be absolute** — cloudflared does not
expand `$HOME` or `~`. Put it in `/etc/cloudflared/`, not in the runtime
user's home.

## 3. Ingress rules

Lumio's footprint on `jarvis` (per current decisions):

| Service        | Local port            | Visibility                                       |
| -------------- | --------------------- | ------------------------------------------------ |
| Kong (API)     | `127.0.0.1:8000`      | **Public** → `app.lumio.example.com`             |
| Frontend       | `127.0.0.1:3000`*     | **Public** → `lumio.example.com`                 |
| Supabase Studio| `127.0.0.1:3001`*     | **Admin** → Tailscale only (do NOT route here)   |
| Postgres       | `127.0.0.1:5432`      | **Admin** → Tailscale only                       |

(* Adjust to actual ports in the lumio compose file.)

### 3.1 `/etc/cloudflared/config.yml`

```yaml
tunnel: 00000000-0000-0000-0000-000000000000   # replace with real UUID
credentials-file: /etc/cloudflared/00000000-0000-0000-0000-000000000000.json

# No metrics on a public port — bind to localhost only
metrics: 127.0.0.1:36500

# Sensible defaults; per-rule originRequest can override
originRequest:
  connectTimeout: 30s
  tlsTimeout: 10s
  tcpKeepAlive: 30s
  keepAliveConnections: 100
  keepAliveTimeout: 90s
  noHappyEyeballs: false
  disableChunkedEncoding: false
  # Critical for §6: forces Access JWT verification at the edge of cloudflared
  # before traffic ever hits the local service.
  # (Set per-rule when the rule is Access-protected.)

ingress:
  # ---- Public surfaces (no Access) ----
  - hostname: app.lumio.example.com
    service: http://127.0.0.1:8000     # Kong
    originRequest:
      httpHostHeader: app.lumio.example.com
      noTLSVerify: false

  - hostname: lumio.example.com
    service: http://127.0.0.1:3000     # Frontend

  # ---- Admin surface that MUST sit behind Access if exposed at all ----
  # Prefer Tailscale (§5); only enable this block if you must reach
  # Studio from a non-tailnet device.
  - hostname: studio.lumio.example.com
    service: http://127.0.0.1:3001
    originRequest:
      access:
        required: true
        teamName: <your-team>          # e.g. "antonio"
        audTag:
          - <APPLICATION_AUD_TAG>      # from the Access app

  # ---- Mandatory catch-all ----
  - service: http_status:404
```

Key rules:

- Matching is **top-to-bottom**. Put the catch-all last.
- A catch-all is mandatory — if cloudflared gets traffic for a hostname
  with no rule, the connection fails opaquely. `http_status:404` makes
  the "didn't match anything" case explicit and audit-friendly.
- `cloudflared tunnel ingress validate` checks YAML syntax;
  `cloudflared tunnel ingress rule <url>` shows which rule a URL hits.
- For zero-downtime config changes, use the **replica pattern**: start a
  second cloudflared with the new config, wait until it's connected,
  stop the first. Same tunnel UUID, two processes.

### 3.2 DNS

Each hostname needs a DNS CNAME pointing at the tunnel:

```bash
cloudflared tunnel route dns jarvis-tunnel app.lumio.example.com
cloudflared tunnel route dns jarvis-tunnel lumio.example.com
# Only if you actually expose Studio publicly:
cloudflared tunnel route dns jarvis-tunnel studio.lumio.example.com
```

These are proxied (orange-cloud) CNAMEs to `<UUID>.cfargotunnel.com`.
This requires the `cert.pem` from §1.4 to be present temporarily.

## 4. Cloudflare Access — gating the admin surface

### 4.1 Application setup

Zero Trust → Access controls → Applications → **Self-hosted and
private** → Add public hostname → `studio.lumio.example.com`.

- Identity providers: enable **One-Time PIN** (built-in, no IdP setup).
  For a single dev this is the minimum. OTP is **single-factor by
  itself** (knowledge of inbox), so layer it with a hardware key
  policy if Studio gives DB-level write access — see §4.2.
- Session duration: **24 h max** for admin (default is 24 h; do not
  bump to a week).
- App Launcher: **off** for admin apps (don't advertise them in the
  Cloudflare app launcher).

### 4.2 Policies — email allowlist + step-up

A practical admin policy bundle:

```jsonc
// Policy 1: Allow Antonio, require fresh login + warm key
{
  "name": "admin-allow-antonio",
  "decision": "allow",
  "include": [
    { "email": { "email": "toto.castaldi@gmail.com" } }
  ],
  "require": [
    // optional: only if you've registered a security key under the IdP
    // { "auth_method": { "auth_method": "swk" } }
  ],
  "session_duration": "8h",
  "purpose_justification_required": true,
  "purpose_justification_prompt": "Reason for accessing Lumio Studio?"
}
```

Notes:

- `include` = "may apply"; `require` = AND-ed conditions; `exclude` =
  carve-outs. All Access apps are **deny-by-default**.
- `purpose_justification_required` writes the justification to the
  Access audit log — useful one-person change journal.
- Service tokens (§4.3) and human policies coexist on the same app: a
  request matching **either** is allowed.

### 4.3 Service tokens for headless access

For backups, CI, or scripted Studio API calls:

1. Zero Trust → Access → Service Auth → Service Tokens → **Create**.
   Set TTL to 1y max; renewal notification is configurable (default
   1 week prior).
2. Copy the **Client Secret immediately** — it's shown once.
3. Add a Service Auth policy to the protected app:

   ```jsonc
   {
     "name": "svc-backup-runner",
     "decision": "non_identity",   // = "Service Auth" in dashboard
     "include": [
       { "service_token": { "token_id": "<TOKEN_UUID>" } }
     ]
   }
   ```

4. From the runner:

   ```bash
   curl -H "CF-Access-Client-Id: <ID>" \
        -H "CF-Access-Client-Secret: <SECRET>" \
        https://studio.lumio.example.com/api/...
   ```

5. Store secrets in the host's secret store (sops / pass / 1Password
   CLI) — never commit them, never put them in env files under
   `/home/toto/lumio/`.

### 4.4 Origin-side JWT validation (defence in depth)

Even with `access.required: true` in cloudflared ingress, validate the
`Cf-Access-Jwt-Assertion` header on the origin if the origin is a
custom app you control:

- `iss` claim must be `https://<team>.cloudflareaccess.com`.
- `aud` claim must match the Application AUD tag.
- JWKS endpoint: `https://<team>.cloudflareaccess.com/cdn-cgi/access/certs`.
- Reject if header missing or signature invalid.

Studio is a third-party app, so the practical defence here is: (a)
require `access.required: true` in the cloudflared rule so cloudflared
itself drops un-JWT'd traffic, and (b) bind Studio to `127.0.0.1` so
nothing other than cloudflared can reach it on the LAN.

## 5. Tailscale overlay for admin

### 5.1 When to prefer Tailscale over CF Tunnel + Access

| Trait                                  | CF Tunnel + Access            | Tailscale                      |
| -------------------------------------- | ----------------------------- | ------------------------------ |
| TLS termination                        | At Cloudflare edge (plaintext visible to CF) | End-to-end WireGuard, no MITM by Tailscale (control plane only handles key exchange) |
| Auth model                             | Browser SSO/OTP/service token | Device + user identity in tailnet |
| Best for                               | Public web traffic, browser SSO | Admin tools, SSH, databases, Studio |
| Latency                                | +1 CF hop                     | Direct peer (DERP fallback)    |
| Failure mode                           | CF outage = no access         | Tailnet outage = no access (control plane only; existing peers keep talking) |

**Rule of thumb:** if a service handles secrets, DB writes, or
infrastructure mutation (Studio, Postgres, Kong admin API, Grafana,
Portainer), **don't put it on the tunnel at all**. Tailscale-only.

### 5.2 Coexistence on `jarvis`

The two daemons don't conflict:

- `tailscaled.service` — installs the `tailscale0` interface.
- `cloudflared.service` — opens outbound QUIC/HTTPS to CF edge.

Both bind localhost-reachable services to `127.0.0.1` and let each
overlay decide what's reachable:

```
Studio (127.0.0.1:3001) ─┬─ NOT in cloudflared ingress
                          └─ reachable via http://jarvis.<tailnet>.ts.net:3001
```

Use MagicDNS so you don't hardcode tailnet IPs. Optional: enable
Tailscale's HTTPS feature so Studio's URL becomes
`https://jarvis.<tailnet>.ts.net` with a real cert (LetsEncrypt via
the Tailscale ACME flow).

### 5.3 Belt-and-braces: Tailscale + Access

If you really want browser access to Studio from a phone without
Tailscale installed:

1. Keep Studio on `127.0.0.1`.
2. Add the cloudflared ingress rule with `access.required: true`.
3. Require both **OTP** AND a hardware-key auth method in the Access
   policy.
4. Set session duration ≤ 4 h, purpose justification on.

This is the only case where the tunnel touches an admin surface.

## 6. Hardening

### 6.1 Service user

The packaged unit runs cloudflared as **root**. Cloudflared needs no
privileged operations for outbound QUIC + loopback HTTP, so run it as
a dedicated user:

```bash
sudo useradd --system --no-create-home --shell /usr/sbin/nologin cloudflared
sudo chown -R root:cloudflared /etc/cloudflared
sudo chmod 0750 /etc/cloudflared
sudo chmod 0640 /etc/cloudflared/*.json /etc/cloudflared/config.yml
```

`systemctl edit cloudflared` and add:

```ini
[Service]
# Drop root
User=cloudflared
Group=cloudflared
AmbientCapabilities=
CapabilityBoundingSet=
NoNewPrivileges=yes

# Filesystem
ProtectSystem=strict
ProtectHome=yes
ReadOnlyPaths=/etc/cloudflared
PrivateTmp=yes
PrivateDevices=yes
ProtectKernelTunables=yes
ProtectKernelModules=yes
ProtectKernelLogs=yes
ProtectControlGroups=yes
ProtectClock=yes
ProtectHostname=yes
LockPersonality=yes
RestrictRealtime=yes
RestrictSUIDSGID=yes
MemoryDenyWriteExecute=yes
SystemCallArchitectures=native
SystemCallFilter=@system-service
SystemCallFilter=~@privileged @resources @mount @debug @cpu-emulation @obsolete

# Network: outbound to CF + loopback only
RestrictAddressFamilies=AF_UNIX AF_INET AF_INET6
IPAddressDeny=any
IPAddressAllow=localhost
IPAddressAllow=link-local
# Cloudflare edge address space (broad; tighten if you map specific colos)
IPAddressAllow=0.0.0.0/0
```

> Trade-off: cloudflared **must** reach the public internet to register
> with CF edge, so `IPAddressDeny=any + IPAddressAllow=0.0.0.0/0` ends
> up effectively open. The other directives still meaningfully reduce
> blast radius (no `/home`, no kernel surfaces, no new privs, syscall
> filter).

> If you later move cloudflared into Docker, run `--user 65532:65532`,
> `--read-only`, drop all caps; same idea.

### 6.2 Don't leak the tunnel UUID

Tunnel UUIDs by themselves aren't secrets (CF deems the credentials
JSON the secret), but they appear in logs, screenshots, and the public
README. Two cheap mitigations:

- Run with `--loglevel info` (default), not `debug`, in production.
- Add `metrics: 127.0.0.1:36500` (as in §3.1) so the Prometheus
  endpoint isn't on `0.0.0.0`.
- Add to `/etc/cloudflared/config.yml`:
  ```yaml
  no-autoupdate: false    # let CF push fixes
  protocol: quic          # default; document explicitly
  ```
- Scrub the UUID from `README.md`, screenshots, and any committed
  config. The actual secret (`<UUID>.json`) lives at `0640` under
  `/etc/cloudflared/` and is never committed.

### 6.3 Prevent the catch-all from being an Access bypass

A subtle failure mode: you add `studio.lumio.example.com` with
`access.required: true`, then later add a new public hostname `foo.com`
to the **tunnel** but forget the corresponding Access app. Cloudflare
serves it without an Access challenge.

Mitigations:

- **Tunnel-side**: never put an admin hostname in cloudflared without
  `access.required: true` on that rule. The catch-all stays
  `http_status:404` — never proxy unknown hostnames to a real backend.
- **Zone-side**: enable a Cloudflare WAF rule that returns 403 for any
  `studio.*` / `admin.*` subdomain that lacks a matching Access app
  (custom rule using `cf.access.exists`).
- **Origin-side**: bind admin services to `127.0.0.1` only. If
  cloudflared (or anything else) goes wrong, the LAN can't reach them.
- **Audit**: `cloudflared tunnel ingress validate` and
  `cloudflared tunnel ingress rule https://studio.lumio.example.com`
  in CI / a pre-commit check — fails the change if the studio host
  resolves to a non-Access rule.

### 6.4 Update hygiene

- Enable `unattended-upgrades` for the `pkg.cloudflare.com` origin —
  `cloudflared` ships frequent security patches.
- Subscribe to `cloudflare/cloudflared` GitHub releases.
- Re-key the package signing key reminder set for **April 2026**
  (Cloudflare key rotation deadline).

## 7. Quick checklist for `jarvis` v1

- [ ] `cloudflared` installed from pinned `noble` apt suite, deb822 file.
- [ ] `cloudflared tunnel login` run **once**, `cert.pem` archived off-host.
- [ ] Named tunnel `jarvis-tunnel` created; `<UUID>.json` under
      `/etc/cloudflared/`, `0640 root:cloudflared`.
- [ ] `/etc/cloudflared/config.yml` with explicit hostnames + catch-all
      `http_status:404`.
- [ ] systemd unit overridden to run as `cloudflared` user with
      hardening drop-in (§6.1).
- [ ] DNS routes created for each public hostname only.
- [ ] Tailscale installed and Studio + Postgres reachable **only**
      via tailnet.
- [ ] If Studio is exposed via tunnel at all: Access app with OTP +
      `access.required: true` in the ingress rule + origin bound to
      `127.0.0.1`.
- [ ] No tunnel UUID in README / screenshots / committed files.
- [ ] CI check: `cloudflared tunnel ingress validate` on PRs that
      touch `config.yml`.
